# 功能开发技能

这个技能用于处理博客项目的功能开发、bug修复、代码重构和性能优化。

## 激活条件

当用户提出以下需求时激活：

- "修复XX bug"
- "添加XX功能"
- "优化XX性能"
- "重构XX代码"
- 或其他明确的开发需求

**不激活的情况**：

- 纯样式调整（使用 style-optimizer）
- 内容管理（使用 content-manager）
- 写博客文章（使用 blog-writer）

## 项目技术栈

- **框架**: Astro 5.17
- **UI库**: React 19
- **样式**: TailwindCSS 3.4
- **内容**: MDX
- **代码高亮**: Expressive Code
- **部署**: GitHub Pages + Vercel

## 工作流程

### Step 1：需求分析

1. **理解需求**：
   - 如果是 bug，询问复现步骤和预期行为
   - 如果是功能，询问具体需求和使用场景
   - 如果是优化，询问优化目标

2. **确认范围**：
   - 影响哪些文件
   - 是否需要修改配置
   - 是否需要添加依赖

3. **制定方案**：
   - 简要说明实现思路
   - 列出需要修改的文件
   - 预估工作量

**示例输出**：

```
我理解你的需求是：[需求描述]

实现方案：
1. 修改 src/components/Header.astro - 添加XX功能
2. 更新 src/layouts/MainLayout.astro - 调整XX逻辑
3. 测试不同页面的表现

预计修改 2-3 个文件，需要测试主页和博客页面。
```

### Step 2：代码分析

1. **定位相关代码**：
   - 使用 `readCode` 或 `readFile` 读取相关文件
   - 理解现有实现逻辑
   - 识别需要修改的部分

2. **检查依赖关系**：
   - 查看组件引用关系
   - 检查是否影响其他功能
   - 确认是否需要更新类型定义

3. **参考示例**：
   - 查看 `references/` 目录下的示例
   - 参考项目中类似的实现

### Step 3：实现代码

1. **编写代码**：
   - 使用 `editCode` 或 `strReplace` 修改文件
   - 遵循项目代码规范
   - 添加必要的注释

2. **代码规范**：
   - 使用 TypeScript 类型注解
   - 遵循 Astro 组件规范
   - TailwindCSS 类名使用语义化命名
   - React 组件使用函数式组件

3. **错误处理**：
   - 添加适当的错误处理
   - 考虑边界情况
   - 提供友好的错误提示

**参考脚本**：

- `scripts/check-build.sh` - 检查构建是否成功
- `scripts/test-feature.sh` - 测试功能是否正常

### Step 4：测试验证

1. **本地构建测试**：

   ```bash
   npm run build
   ```

   - 检查是否有构建错误
   - 查看构建输出

2. **类型检查**（如果修改了 TypeScript）：
   - 使用 `getDiagnostics` 检查类型错误
   - 修复所有类型问题

3. **功能测试**：
   - 测试新功能是否正常工作
   - 测试是否影响现有功能
   - 测试不同设备和浏览器

4. **检查清单**：
   - [ ] 构建无错误
   - [ ] 类型检查通过
   - [ ] 功能正常工作
   - [ ] 没有破坏现有功能
   - [ ] 代码符合规范

### Step 5：用户确认

1. **展示修改**：

   ```
   我已经完成了以下修改：

   修改的文件：
   - src/components/Header.astro - 添加了XX功能
   - src/layouts/MainLayout.astro - 优化了XX逻辑

   主要改动：
   1. [改动描述]
   2. [改动描述]

   测试结果：
   - ✅ 构建成功
   - ✅ 功能正常
   - ✅ 没有破坏现有功能
   ```

2. **询问反馈**：
   - 功能是否符合预期？
   - 是否需要调整？
   - 是否需要添加其他功能？

3. **根据反馈修改**：
   - 如果不满意，询问具体问题
   - 修改后重新测试
   - 直到用户满意

4. **检查中英文版本**（如果涉及页面内容）：
   - 如果修改了页面文本内容，确认是否需要英文版本
   - 如果修改了关于页面、个人介绍等，提供英文翻译
   - 确保中英文版本内容一致

### Step 6：Git 提交

**用户确认满意后**：

1. **暂存修改**：

   ```bash
   git add [修改的文件]
   ```

2. **提交代码**：

   ```bash
   git commit -m "feat: [功能描述]"
   # 或
   git commit -m "fix: [bug描述]"
   ```

3. **推送代码**：

   ```bash
   git push
   ```

4. **确认部署**：
   - 告知用户代码已推送
   - Vercel 会自动部署
   - 提供预计部署时间（通常 1-2 分钟）

## 提交信息规范

遵循 Conventional Commits 规范：

- `feat:` - 新功能
- `fix:` - bug 修复
- `refactor:` - 代码重构
- `perf:` - 性能优化
- `style:` - 代码格式调整
- `docs:` - 文档更新
- `chore:` - 构建配置等

**示例**：

```
feat: 添加文章搜索功能
fix: 修复主题切换在博客页面不同步的问题
refactor: 重构 Header 组件逻辑
perf: 优化图片加载性能
```

## 常见开发场景

### 场景 1：修复 Bug

**示例**：主题切换不同步

1. 复现 bug
2. 定位问题代码（MainLayout.astro）
3. 分析原因（localStorage 读取时机）
4. 修复代码
5. 测试验证
6. Git 提交

### 场景 2：添加新功能

**示例**：添加文章搜索

1. 确认需求（搜索标题、标签、内容）
2. 设计方案（客户端搜索 vs 服务端搜索）
3. 创建组件（SearchBar.astro）
4. 实现搜索逻辑
5. 集成到页面
6. 测试验证
7. Git 提交

### 场景 3：性能优化

**示例**：优化图片加载

1. 分析性能瓶颈
2. 制定优化方案（懒加载、压缩）
3. 实现优化
4. 测试性能提升
5. Git 提交

### 场景 4：代码重构

**示例**：重构组件逻辑

1. 识别需要重构的代码
2. 制定重构方案
3. 逐步重构
4. 确保功能不变
5. 测试验证
6. Git 提交

### 场景 5：多语言内容更新

**示例**：更新关于页面

1. 确认需要更新的内容
2. 修改中文版本（`src/pages/about.astro` 或对应文件）
3. 创建或更新英文版本（`src/pages/en/about.astro`）
4. 确保中英文内容一致
5. 测试两个版本
6. Git 提交

**需要双语版本的页面**：

- 关于页面（About）
- 主页个人介绍
- 导航链接文本
- 页脚信息
- 错误页面（404等）

## 注意事项

### 必须遵守的规则

1. **不破坏现有功能**：
   - 修改前先理解现有逻辑
   - 测试是否影响其他功能
   - 保持向后兼容

2. **遵循项目规范**：
   - 代码风格保持一致
   - 使用项目已有的工具和库
   - 不随意添加新依赖

3. **充分测试**：
   - 本地构建测试
   - 功能测试
   - 回归测试

4. **清晰的提交信息**：
   - 使用规范的提交格式
   - 描述清楚修改内容
   - 必要时添加详细说明

5. **中英文双语支持**：
   - 修改页面内容时，同时更新中英文版本
   - 修改个人介绍、关于页面等静态内容时，提供英文翻译
   - 确保中英文版本内容一致
   - 技术术语保持统一

### 开发最佳实践

1. **小步快跑**：
   - 一次只做一件事
   - 频繁提交
   - 及时测试

2. **代码可读性**：
   - 添加必要的注释
   - 使用语义化命名
   - 保持代码简洁

3. **性能考虑**：
   - 避免不必要的重渲染
   - 优化资源加载
   - 考虑移动端性能

4. **用户体验**：
   - 提供友好的错误提示
   - 考虑加载状态
   - 保持交互流畅

## 常见问题处理

### Q: 构建失败怎么办？

A:

1. 查看错误信息
2. 检查语法错误
3. 检查类型错误
4. 检查依赖是否安装
5. 清除缓存重新构建

### Q: 功能在本地正常但部署后异常？

A:

1. 检查环境变量
2. 检查路径是否正确（相对路径 vs 绝对路径）
3. 检查构建输出
4. 查看 Vercel 部署日志

### Q: 不确定如何实现某个功能？

A:

1. 查看项目中类似的实现
2. 查看 Astro 官方文档
3. 参考 references/ 目录下的示例
4. 询问用户是否有参考案例

### Q: 修改后影响了其他功能？

A:

1. 回滚修改
2. 重新分析影响范围
3. 采用更保守的实现方案
4. 充分测试后再提交
