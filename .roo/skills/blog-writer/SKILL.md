# 博客写作技能

将知识库笔记转化为符合用户风格的博客文章。

## 激活条件

当用户提出以下需求时激活：

- "转化这篇笔记"
- "写成博客"
- "发布文章"
- "把XX笔记改成博客"
- "激活文字转化"
- 或其他明确表示要将笔记转化为博客的请求

**不激活的情况**：

- 简单问候
- 询问进度
- 修改已有博客的小调整

## 核心工作流程

### Step 1：检查知识库和转化状态

1. **读取转化跟踪文件**：

   ```bash
   # 查看转化状态（可选，如果 jq 可用）
   bash .kiro/skills/blog-writer/scripts/update-tracker.sh status
   bash .kiro/skills/blog-writer/scripts/update-tracker.sh list-pending
   ```

   如果脚本不可用，直接读取 `conversion-tracker.json` 文件。

2. **扫描知识库目录**：
   - 扫描 `知识库/02_Notes/` 目录
   - 排除已转化的笔记
   - 排除已跳过的笔记
   - 识别新的可转化笔记

3. **列出候选笔记**：
   - 优先展示 pending 列表中的高优先级笔记
   - 列出新发现的可转化笔记
   - 询问用户想转化哪一篇

### Step 2：通过提问了解知识掌握程度

读取目标笔记后，**不要直接转化**，而是通过提问来了解用户对这个知识的真实掌握程度。

**提问策略**（3-5 个问题）：

1. **概念理解类**：
   - "你能用自己的话解释一下XX是什么吗？"
   - "你觉得XX解决了什么问题？"

2. **实践经验类**：
   - "你在项目中用过XX吗？遇到过什么坑？"
   - "你是在什么场景下接触到XX的？"

3. **深度思考类**：
   - "你觉得XX的优缺点是什么？"
   - "XX和YY相比，你觉得哪个更好？为什么？"

4. **类比创造类**：
   - "如果让你用生活中的例子来解释XX，你会怎么比喻？"

**根据回答判断掌握程度**：

- **深度理解**（可以直接转化）：
  - 能清晰解释概念
  - 有实际项目经验或深入研究
  - 能指出技术的优缺点和适用场景
  - 能创造生动的类比

- **中等理解**（需要补充内容）：
  - 理解基本概念，但缺少实践经验
  - 能解释原理，但不够深入
  - 需要通过对话补充更多细节

- **浅层了解**（建议暂缓转化）：
  - 只知道定义，说不清楚原理
  - 没有个人见解
  - 无法用自己的话解释

### Step 3：转化为博客

根据用户的回答和笔记内容，转化为博客文章。

**文章结构**（参考 `references/writing-style-guide.md`）：

```markdown
---
title: "标题（简洁直白）"
description: "一句话描述，通常以'说白了'开头"
author: HUTAO667
tags: ["标签1", "标签2", "标签3"]
pubDate: "YYYY-MM-DD"
lang: "zh" # 或 "en"
---

【开头段落：从实际问题切入】
我在XXX的时候遇到一个问题：...

# 什么是XX

【直接定义，一两句话说清楚】

# 用类比来理解

【用生活中的例子类比，优先使用用户自己创造的类比】

# XX的工作原理/流程

【分步骤讲解，用step1、step2或小标题】

# 在我项目中的应用（可选）

【分享实际项目经验，如果有的话】

# 我的看法

【个人理解和思考】

# 总结

说白了，XX就是...

---

**参考资料**：

- [链接1](url) - 描述
- [链接2](url) - 描述
```

**转化要点**：

1. **获取原笔记创建时间**：
   - 使用文件的创建时间作为 pubDate
   - 不要使用当天日期
   - Windows: 使用 `Get-Item` 查看 CreationTime

2. **保持口语化**：
   - 多用"说白了"、"其实就是"
   - 用"我觉得"、"我在项目里"
   - 避免学术化表达

3. **使用用户的类比**：
   - 优先使用用户在回答中创造的类比
   - 如果没有，可以根据概念创造合适的类比
   - 类比要贴近生活，容易理解

4. **体现真实经历**：
   - 用户提到的实践经验要详细展开
   - 用户看的论文要提到
   - 用户承认"没用过"要如实写出来

5. **添加参考资料**：
   - 技术概念类文章必须添加参考资料
   - 包括官方文档、论文、相关文章
   - 每个链接都要有简短描述

6. **创建中英文版本**：
   - 先创建中文版本
   - 然后翻译为英文版本
   - 保持技术术语的准确性
   - 保持文章结构和风格

### Step 4：展示并询问反馈

1. **展示转化结果**：
   - 完整展示转化后的文章
   - 说明做了哪些调整

2. **询问反馈**：

   ```
   我已经把笔记转化成博客了，主要做了这些调整：
   - 补充了开头的问题引入
   - 使用了你提到的"XX"类比
   - 扩展了你在项目中的应用
   - 添加了参考资料

   你觉得怎么样？有需要修改的地方吗？
   ```

3. **根据反馈修改**：
   - 如果用户不满意，询问具体哪里需要改
   - 修改后再次确认
   - 直到用户满意

### Step 5：验证和构建检查

用户确认满意后，进行验证和构建检查。

**5.1 快速验证（可选）**

使用验证脚本快速检查 frontmatter 格式：

```bash
# 验证中文版本
bash .kiro/skills/blog-writer/scripts/validate-post.sh "src/blog/2026/M/文章名.mdx"

# 验证英文版本
bash .kiro/skills/blog-writer/scripts/validate-post.sh "src/blog/2026/M/Article-Name.mdx"
```

这个脚本会检查：

- title、description、author、pubDate、tags、lang 字段是否存在
- author 是否为 HUTAO667
- pubDate 格式是否正确（YYYY-MM-DD）
- lang 是否为 zh 或 en

如果脚本不可用，可以跳过这一步，直接运行构建。

**5.2 运行构建**

```bash
npm run build
```

**检查要点**：

- frontmatter 格式是否正确
- 是否有语法错误
- 数学公式是否正确（注意：MDX 可能不支持 LaTeX，需要改用文字描述）
- 如果有错误，修复后重新构建

**常见构建错误**：

1. **缺少 author 字段**：

   ```
   author: Required
   ```

   解决：添加 `author: HUTAO667`

2. **pubDate 格式错误**：

   ```
   pubDate: Invalid date format
   ```

   解决：使用 `"YYYY-MM-DD"` 格式

3. **数学公式错误**：
   ```
   Attention is not defined
   ```
   解决：将 LaTeX 公式改为文字描述或使用行内格式

### Step 6：更新转化跟踪

```bash
# 如果 jq 可用
bash .kiro/skills/blog-writer/scripts/update-tracker.sh add-converted \
  "知识库/02_Notes/笔记名.md" \
  "src/blog/2026/M/文章名.mdx"
```

如果脚本不可用，手动编辑 `conversion-tracker.json`：

```json
{
  "notePath": "知识库/02_Notes/笔记名.md",
  "blogPath": "src/blog/2026/M/文章名.mdx",
  "blogPathEn": "src/blog/2026/M/Article-Name.mdx",
  "convertedDate": "YYYY-MM-DD",
  "status": "published",
  "hasEnglish": true
}
```

同时从 `pending` 列表中移除该笔记。

### Step 7：Git 提交和推送

```bash
git add .
git commit -m "新增博客：标题（中英文）"
git push
```

确认推送成功后，告知用户：

- 文章已推送（中英文版本）
- Vercel 会自动部署
- 转化跟踪已更新

## 重要规则

### 必须遵守

1. **不转化商业机密**：
   - 如果笔记涉及公司项目细节，跳过
   - 如果涉及未公开的技术，询问用户

2. **保持真实性**：
   - 不编造用户没有的经验
   - 不确定的内容要询问用户
   - 用户承认"没用过"要如实写出来

3. **尊重原笔记**：
   - 不改变用户的核心观点
   - 保留用户的个人风格

4. **检查知识准确性**：
   - 如果发现笔记中有明显错误，提醒用户
   - 技术概念要准确

### 可以灵活处理

1. **补充类比**：
   - 如果原笔记没有类比，可以创造合适的类比
   - 但要符合用户的表达风格

2. **扩展结构**：
   - 原笔记可能比较简略，可以适当扩展
   - 但不要偏离原意

3. **优化表达**：
   - 补充标点符号
   - 调整句子结构使其更流畅
   - 但保持口语化风格

## 常见问题

### Q: 笔记内容太少怎么办？

A: 通过 Step 2 的提问了解更多细节，或者建议用户补充内容后再转化。

### Q: 笔记中有图片怎么办？

A: 检查图片是否在 `public/blog-images/` 目录，如果在，保持引用；如果不在，询问用户。

### Q: 不确定用户对某个概念的理解程度？

A: 这就是 Step 2 的作用，通过提问来了解。

### Q: 参考资料找不到？

A: 使用 `remote_web_search` 搜索，或询问用户是否有推荐。

### Q: 用户想修改已发布的博客？

A: 这不属于本技能范围，直接修改对应的 mdx 文件即可。

### Q: 脚本无法运行怎么办？

A: 可以手动操作，参考 `references/scripts-guide.md` 中的替代方案。

## 成功标准

一篇成功转化的博客应该：

- ✅ 保持用户的口语化风格
- ✅ 有清晰的问题引入
- ✅ 有生动的类比说明（优先使用用户创造的）
- ✅ 有个人实践经验（如果原笔记有）
- ✅ 有"我的理解"章节
- ✅ 有"说白了"开头的总结
- ✅ 有完整的参考资料（技术类文章）
- ✅ 构建无错误
- ✅ 成功推送到 Git
- ✅ 转化跟踪已更新
